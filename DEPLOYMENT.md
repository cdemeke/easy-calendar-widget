# Deployment Guide

This document covers how to release Easy Calendar Widget — from cutting a release to what happens behind the scenes in CI.

## Prerequisites

Before your first release, ensure these GitHub repository secrets are configured:

| Secret | Description |
|--------|-------------|
| `CERTIFICATE_P12` | Base64-encoded Developer ID Application certificate (.p12) |
| `CERTIFICATE_PASSWORD` | Password used when exporting the .p12 |
| `APPLE_ID` | Apple ID email for notarization |
| `APPLE_ID_PASSWORD` | App-specific password (not your Apple ID password) |
| `APPLE_TEAM_ID` | Your Apple Developer Team ID (`TQV66JM3KL`) |

### Generating the secrets

1. **Export your Developer ID certificate** from Keychain Access as a `.p12` file.
2. **Base64-encode it**:
   ```bash
   base64 -i DeveloperID.p12 | pbcopy
   ```
   Paste the result into the `CERTIFICATE_P12` secret.
3. **Create an app-specific password** at [appleid.apple.com](https://appleid.apple.com) under Sign-In and Security > App-Specific Passwords. Use this for `APPLE_ID_PASSWORD`.

## Releasing a new version

The entire release process is automated via GitHub Actions. To deploy:

```bash
# 1. Make sure you're on main with a clean working tree
git checkout main
git pull

# 2. Tag the release (triggers the workflow)
git tag v1.1.0
git push origin v1.1.0
```

That's it. Pushing a tag matching `v*` triggers the CI pipeline.

## What the CI pipeline does

The workflow (`.github/workflows/release.yml`) runs on `macos-latest` and performs these steps in order:

1. **Checkout & Xcode setup** — uses the latest stable Xcode on the runner.
2. **Install signing certificate** — decodes the `.p12` from secrets into a temporary keychain.
3. **Build Release archive** — runs `xcodebuild archive` with:
   - `-configuration Release` (no "Dev" labels, production bundle ID)
   - `CODE_SIGN_IDENTITY="Developer ID Application"`
   - `ENABLE_HARDENED_RUNTIME=YES`
4. **Export .app** — copies the built app out of the archive.
5. **Notarize the app** — submits to Apple via `xcrun notarytool`, waits for approval, staples the ticket.
6. **Create styled DMG** — uses `create-dmg` with a generated background, app icon, and Applications shortcut.
7. **Notarize the DMG** — submits the DMG itself for notarization and staples it.
8. **Publish GitHub Release** — uploads the DMG and auto-generates release notes from commits since the last tag.

## Building a DMG locally

For local testing of the DMG packaging (without notarization):

```bash
# Install create-dmg if you haven't already
brew install create-dmg

# Run the local script
scripts/create_dmg.sh
```

This builds a Release archive, exports the app, and packages it into a DMG. It does not sign with Developer ID or notarize — use this only for local verification.

## Debug vs Release builds

The project is configured so that Debug and Release are fully separated:

| | Debug | Release |
|---|---|---|
| Bundle ID | `com.chrisdemeke.easycalendarwidget.dev` | `com.chrisdemeke.easycalendarwidget` |
| App name | EasyCalendarWidget - Dev | EasyCalendarWidget |
| Widget names | "Dev - Calendar", "Dev - 6-Month Calendar" | "Calendar", "6-Month Calendar" |
| Code signing | Apple Development (automatic) | Developer ID Application (manual in CI) |

This means you can have both Debug and Release builds installed side-by-side on the same machine without conflicts.

## Versioning

Update the version in Xcode before tagging:

1. Select the project in the navigator
2. Under both targets (app + widget extension), update **Marketing Version** (e.g., `1.1.0`)
3. Xcode auto-increments **Current Project Version** on archive, or bump it manually

Then commit, tag, and push.

## Troubleshooting

### Notarization fails
- Check the notary log: the workflow automatically fetches it on failure.
- Common causes: missing hardened runtime, unsigned frameworks, restricted entitlements.
- Ensure your Developer ID certificate hasn't expired.

### Code signing errors in CI
- Verify `CERTIFICATE_P12` is up to date (certificates expire annually).
- Re-export and re-encode if the password changed.

### DMG looks wrong
- The background is generated by `scripts/generate_dmg_background.py` at build time.
- Window size and icon positions are set in the workflow — adjust the `create-dmg` flags if needed.
